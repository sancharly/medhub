# MedHub goals

This document describes high level goals and technical constraints that can be used to extract User needs and non functional requirements. They shall be used as a starting point to start the design specification of the application.

## Goals

| Goal                                | Description                                                                                                                                                                                                                                                                                                                          |
|-------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Web application                     | The application shall have a web interface accessible from a web explorer                                                                                                                                                                                                                                                            |
| User types                          | The application shall consider the following user types: doctor, patient, system administrator, Administrative personal. Different user types can access different data and modules                                                                                                                                                  |
| User profiles                       | Each user shall be able to access its profile with a username (e-mail) and a password                                                                                                                                                                                                                                                |
| User groups                         | User groups can be defined by the system administrator. Users can be added to groups automatically (by user type, for example) or manually                                                                                                                                                                                           |
| Doctors permissions                 | Doctors can access the clinical data of their own patients, but not others doctors' patients                                                                                                                                                                                                                                         |
| Patient permissions                 | Patients can access they own personal and clinical data. They can give or revoke clinical data access to doctors.                                                                                                                                                                                                                    |
| Administrative personal permissions | Administrative personal can access limited personal information from patient and doctors (Name, Family name, date of birth) but no clinical data.The can schedule anointments between doctors and patients                                                                                                                           |
| Modular approach                    | The application has a modular architecture so new functionality can be added through plugins. PLugins can define new modules that can be enabled for a subset of users by a system administrator. Modules enable new functionality like DICOM viewer, surgery planning or other features to be defined in the future.                |
| Modules access                      | Doctors can access the modules that have been enabled by the system administrator to their user group                                                                                                                                                                                                                                |
| Clinical entry                      | Doctors can add a clinical entry associated with a patient. The clinical entry has, at least, the following fields: Date and time, Patient,  Doctor, Description, attached files (DICOM images, reports, analysis results, etc..) or other fields defined by                                                                         |
| DICOM viewer Module                 | The first working prototype shall have a DICOM viewer module to demonstrate the plugins system. This module shall allow users to open DICOM images (CT scans, MRI, X rays images...) and visualize them. Minimum controls shall include level/contrast manipulation, zoom in/out, slice navigation for volumetric rendering and pan. |

## Constraints

| Constraint                | Description                                                                                                                         |
|---------------------------|-------------------------------------------------------------------------------------------------------------------------------------|
| Explorer compatibility    | The application shall be compatible with all major web explorers (Safari, Firefox, Chrome, Edge)                                    |
| FHIR standard             | Data governance shall comply with FHIR (Fast Healthcare Interoperability Resources) for standardization and future interoperability |
| Devices compatibility     | The application shall be accessible from desktop navigators as well as from mobile  devices (tablets and phones).                   |
| GDPR                      | The application shall be compliant with GDPR laws in terms of data storage and treatment                                            |
| Medical device regulation | the application shall comply with EY and USA medical devices regulations (MDR and FDA)                                              |
| IEC 62304                 | The applications shall be developed in accordance with IEC 62304 standard                                                           |
| Cyber security            | The application shall comply with cybersecurity standards and guidances that apply to medical device software in Europe and the USA |
